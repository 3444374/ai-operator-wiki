# 精读笔记：Cortex AISQL (SIGMOD 2026)

**论文**：Aggarwal, Chen, Datta, Han et al. *Cortex AISQL: A Production SQL Engine for Unstructured Data.* SIGMOD Companion 2026, arXiv:2511.07663

**来源级别**：CCF-A 会议论文 + 产业生产系统

**阅读日期**：2026-07-15

---

## 一句话核心结论

Snowflake 在生产环境中将六类 AI 算子（EMBED/COMPLETE/FILTER/CLASSIFY/JOIN/AGG）作为 SQL 执行引擎一等公民，通过 AI 感知查询优化、模型级联和语义 Join 重写三项技术，在 2025 年 7-9 月生产监控中证实 AI 算子主导查询成本，约 40% 查询涉及多表操作。

---

## 解决的问题

传统 SQL 引擎无法处理非结构化数据上的语义操作（semantic operations）。AI 推理调用比传统 SQL 算子昂贵数个量级，延迟和吞吐特征不同，编译期成本和选择率未知。简单地把 AI 函数作为 UDF 嵌入 SQL，会导致查询计划质量灾难性下降。

---

## 核心技术

### 1. AI 感知查询优化（2-8× 加速）

**问题**：传统优化器无条件将谓词下推到靠近扫描的位置，但 AI 谓词（AI_FILTER）的每行推理成本极高——对百万行表盲目下推会导致数十万次 LLM 调用。

**方案**：将 LLM 推理成本作为一阶优化目标。代价模型：`C_op(n) = n × c_model + α`（每行 GPU 成本 + 固定开销）。优化器在 `w1·C_LLM + w2·C_CPUIO` 联合代价下选择最优计划。必要时将昂贵 AI 谓词**上拉**到 Join 之后，先用传统结构化过滤缩小数据集。

**关键案例**：一次涉及 papers 和 paper_images 两表 Join 的查询中，原始计划触发 **~110,000 次模型调用**；AI 感知优化后将 AI 谓词上拉到 Join 后，仅需 **330 次调用**——**~300× 改进**。

### 2. 自适应模型级联（2-6× 加速，90-95% 质量保持）

**方案**：小模型（如 Llama 3.1-8B）作为 proxy，处理大部分行；大模型（如 Llama 3.3-70B）作为 oracle，仅处理 proxy 不确定的行。使用重要性采样在运行时学习双阈值路由策略，将行划分到接受区、不确定区和拒绝区。

**效果**：NQ 数据集上 5.85× 加速，F1 接近 oracle 水平。平均节省 65.5% 执行时间。

### 3. 语义 Join 重写（15-70× 加速）

**问题**：`AI_JOIN`（如"找出与这段描述语义匹配的所有图片"）需要 O(N×M) 交叉连接 + AI_FILTER。

**方案**：将二次复杂度的语义 Join 重写为线性多标签分类——将右侧表的列作为标签，AI_JOIN 转换为 AI_CLASSIFY 操作。

**效果**：平均 30.7× 加速。CNN 数据集上从 4.4 小时降至 3.8 分钟（69.5×），F1 平均提升 44.7 个百分点。

## 六大 AI SQL 算子

| 算子 | 语义 | 对应本课题场景 |
|---|---|---|
| `AI_EMBED` | 生成向量 | embedding / RAG ingestion |
| `AI_COMPLETE` | 文本生成 | offline LLM |

| `AI_CLASSIFY` | 分类 | AI predicate |
| `AI_JOIN` | 语义连接 | 多表 AI workload |
| `AI_AGG` / `AI_SUMMARIZE_AGG` | 语义聚合/摘要 | 高级分析 |

## 本论文的不足（对本课题而言）

1. **闭源系统**：内部执行阶段（数据如何从表到 batch、如何调用模型服务、如何写回）不可拆分
2. **无外部执行视角**：Cortex AISQL 的 AI 算子在 Snowflake 内部执行；不研究"数据库触发后经由外部 worker/Ray/GPU 服务再写回"的路径
3. **不关心写回瓶颈**：论文关注查询执行优化（如何减少 LLM 调用次数、如何选择模型），不关注结果持久化阶段
4. **不区分 batch 构造策略**：batch size、partition 粒度、task/actor 调度策略在 Snowflake 内部是隐式的
5. **不与 Ray/Daft/Lance 无关**：Snowflake 使用自研执行引擎，无法为本课题的外部执行链路提供直接机制参考

---

## 对本课题的含义

**强烈支撑课题动机，但不与本课题方法重叠。**

1. **需求证实**：Snowflake 的生产数据直接证实了"数据库 AI 算子执行成本是瓶颈"——这为本课题的动机提供了最强产业证据
2. **方向确认**：AI-aware 优化（3 项技术）说明"感知 AI 算子特征的执行策略"是有效方向，本课题可以在外部执行链路中延续这一思路
3. **互补关系**：Snowflake 做的是"数据库内核 + AI 算子 + 查询优化"，本课题做的是"数据库触发 + 外部执行链路 + 写回"——两者不重复
4. **引用策略**：§2.1 作为产业需求证据，§2.4 作为"内部执行不可见、外部链路研究空白"的论证来源
