# 精读笔记：GaussML (ICDE 2024) + 对照路线分析

**论文**：Li, Sun, Li, Wang, Nie, Xu. *GaussML: An End-to-End In-database Machine Learning System.* ICDE 2024.

**来源级别**：CCF-A 会议论文（华为 + 清华大学）

**阅读日期**：2026-07-15

---

## 一句话核心结论

华为 openGauss 中将 20+ ML 算子以原生 SQL 语法直接集成进查询引擎，替代 ML-as-UDF 方案，配合 ML 感知优化器和 SIMD 加速，比 Apache MADlib 快 2-6×。

---

## 解决了什么问题

传统"ML-as-UDF"方法有两个核心缺陷：
1. **安全风险**：UDF 可能引入漏洞代码
2. **性能瓶颈**：UDF 受限于 SQL 查询算子的数据访问与执行模式约束（无法利用 SIMD、无法做 ML 感知优化）

GaussML 的方案：**把 ML 算子做成数据库原生算子，而不是 UDF**。

---

## 核心技术

1. **原生 SQL 接口**：将典型 ML 算子直接集成进查询引擎——不是 `SELECT my_udf(x)`，而是让优化器理解 ML 算子的语义和代价
2. **ML 感知优化器**：引入 ML 感知的基数与代价估计器，优化 SQL+ML 查询计划
3. **SIMD 与数据预取**：利用 SIMD 加速 ML 算子训练，利用数据预取减少缓存 miss
4. **分布式并行**：利用数据库自身的并行和分布式能力进行训练与推理

---

## 关键结果

- 支持 20+ 常用 ML 算法
- 比 Apache MADlib 快 **2-6×**
- 在 openGauss 数据库中实现

---

## 对照分析：GaussML vs 本课题

| 维度 | GaussML | 本课题 |
|---|---|---|
| 架构路线 | ML 能力嵌入数据库内核 | 数据库触发 → 外部执行 → 写回 |
| ML 算子位置 | 查询引擎内部 | 外部 Ray worker + GPU 模型服务 |
| 优化对象 | 数据库内的算子执行 + SIMD | 外部链路的数据组织 + 调度 + 推理 + 写回 |
| 数据格式 | 数据库内部格式 | Arrow RecordBatch |
| 模型类型 | 传统 ML（分类、回归等） | GPU-backed embedding / LLM |
| 写回方式 | 数据库内部自然写回 | psycopg/pgvector/Lance 显式写回 |

**关键区分**：GaussML 代表"把模型拉进数据库"（DB4AI）——优势是减少数据移动、降低推理延迟，但受限于数据库进程边界内的资源。本课题代表"数据出数据库再回来"（外部执行）——优势是可利用 GPU 集群、Ray 弹性调度、独立模型服务，但引入数据移动和写回开销。

**两种路线适用于不同场景**，不是谁替代谁的问题——开题报告中需明确这个定位。
