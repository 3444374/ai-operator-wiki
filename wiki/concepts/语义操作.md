# 语义操作

## 定义
语义操作（Semantic operations）是指对非结构化数据（文本、图像、音频等）进行语义理解与转换的一类 AI 操作，包括向量嵌入、文本生成、语义分类、过滤和聚合等。它们构成了 Cortex AISQL 中六大 AI SQL 算子的抽象概念基础，通过将大语言模型（LLM）的推理能力融入关系代数，使得传统 SQL 引擎能够像处理标准关系算子一样处理这些原本依赖精确匹配无法实现的语义任务。

- 语义操作具体包括语义嵌入（semantic embedding）、语义过滤（semantic filtering）、语义分类（semantic classification）、语义连接（semantic join）和语义聚合（semantic aggregation）五种基础操作。
## 关键特征
- **高层次抽象**：语义操作不是具体的算子实现，而是对“理解内容含义”这类操作的统一概念抽象，覆盖嵌入、生成、分类、过滤、聚合等多个维度。
- **弥补传统 SQL 的不足**：传统 SQL 依赖精确匹配和结构化条件，无法直接处理文本语义、图像内容等非结构化查询；语义操作通过 LLM 推理桥接了语义鸿沟。
- **融入关系代数**：语义操作被形式化为一种新型数据库操作，可以融入查询树和关系代数模型中，使查询引擎能够像处理 `JOIN`、`FILTER` 一样调度和优化它们。
- **AI 算子基础**：AI 算子（如 [[concepts/ai_embed|ai_embed]]、[[concepts/ai_filter|ai_filter]]、[[concepts/ai_join|ai_join]]）是语义操作在 SQL 中的具体实例化，而“语义操作”这一概念则为这些算子的统一设计、优化和成本建模提供了理论支撑。
- **不可简化为 UDF 的集成深度**：将 AI 能力简单包装为用户定义函数（UDF）注入 SQL 引擎会导致查询计划质量灾难性下降，这正是语义操作必须作为一等公民深度融入代价估计与优化框架的根本原因，而不能仅以“外部调用”的方式存在。
## 应用
- **Cortex AISQL 查询处理**：在 Snowflake 的 AI SQL 系统中，用户的每一个 AI 语义查询（例如“找出与国防相关的段落并连接文档”）都被映射为一棵由语义操作构成的逻辑查询树，然后通过 [[concepts/ai-aware-query-optimization|AI 感知查询优化]] 转化为物理执行计划。
- **成本与性能建模**：语义操作抽象使得 [[concepts/llm-inference-cost-model|LLM 推理成本模型]] 能够统一估算不同操作的资源消耗，从而在优化阶段进行代价比较和选择。
- **语义数据管道**：在 ETL 或数据分析任务中，可将语义操作嵌入到数据管道中，实现自动化分类、情感分析、摘要生成等，而无须外部 AI 服务的手动调用。
